# osctrl auth providers — design spec v0.1

**Scope**: extend osctrl-api + the React SPA with federated identity-provider
login. Build OIDC first; design for SAML as a second provider type, but defer
SAML implementation until customer demand justifies it.

**Build mode**: fork-first on `alvarofraguas/osctrl`. Each layer lands as its
own stacked branch (`auth/00`..`auth/04`) so the work can be reviewed in
slices and selectively offered upstream.

## Decisions locked

| Question | Decision |
|---|---|
| Package layout | Full `pkg/auth/` abstraction now (option a). `Provider` interface + `pkg/auth/oidc/` concrete impl. |
| Config scope | Per-environment from day one. Each env can have its own OIDC config. |
| State storage | JWT-signed state cookie, signed with osctrl-api's existing JWT secret. No new infra. |
| JIT user creation | Match legacy admin: create AdminUser with **zero** env permissions. Operator grants manually. |
| SAML | Architecture supports it cleanly; implementation deferred. |
| Refresh tokens | Skip — osctrl JWT TTL drives re-auth, not the IdP id_token. |
| Single sign-out | Deferred. Add when a customer asks. |
| Multiple IdPs per env | Deferred — unique index `(env_id, type)` reserves the option. |
| Group-driven role mapping | Deferred behind a clean extension point; v1 matches legacy admin behavior. |

## Architecture

### Provider interface

```go
// pkg/auth/provider.go
type Provider interface {
    Type() string
    LoginURL(state State) (string, error)
    HandleCallback(ctx context.Context, r *http.Request) (ResolvedIdentity, error)
}

type ResolvedIdentity struct {
    Subject           string
    PreferredUsername string
    Email             string
    Name              string
    Groups            []string
    Raw               map[string]any
}

type State struct {
    EnvUUID  string  // which env's provider this state belongs to
    Nonce    string  // OIDC nonce
    Verifier string  // PKCE verifier (OIDC only)
}
```

The interface is intentionally minimal. SAML's eventual `HandleCallback`
implementation will need to inspect the POST body (SAML ACS endpoint receives
the assertion via form POST), but the signature accommodates that since the
provider gets the whole `*http.Request`.

### Database

One table, JSON config blob per protocol:

```sql
CREATE TABLE env_auth_providers (
  id              bigserial PRIMARY KEY,
  env_id          bigint NOT NULL REFERENCES tls_environments(id),
  type            text NOT NULL,            -- 'oidc' | 'saml' (future)
  display_label   text NOT NULL,            -- "Acme Corp SSO"
  enabled         bool NOT NULL DEFAULT true,
  config          jsonb NOT NULL,           -- structure varies by type
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now(),
  deleted_at      timestamptz
);
CREATE UNIQUE INDEX idx_env_auth_one_per_env_per_type
  ON env_auth_providers (env_id, type) WHERE deleted_at IS NULL;
```

For OIDC, `config` shape:
```json
{
  "issuer_url": "https://...",
  "client_id": "...",
  "client_secret_encrypted": "<aes-gcm ciphertext, base64>",
  "redirect_url": "https://.../api/v1/auth/oidc/callback",
  "scopes": ["openid", "profile", "email"],
  "username_claim": "preferred_username",
  "groups_claim": "groups",
  "required_groups": ["osctrl-admins"],
  "jit_provision": true,
  "use_pkce": true
}
```

Encryption-at-rest: `client_secret` encrypted with AES-GCM using a key derived
from the api's JWT secret via HKDF (separate domain string so the same secret
can't unlock data it wasn't supposed to). On read, decrypt before passing to
the provider.

### Routes

```
GET  /api/v1/login/{env}/methods             public — lists configured auth methods
GET  /api/v1/auth/{type}/login/{env}         public — kicks off provider flow
GET  /api/v1/auth/{type}/callback            public — provider callback
POST /api/v1/auth/{type}/callback            (SAML later — ACS endpoint takes POST)
```

The methods endpoint is reachable without auth (the user can't have authed
yet). It returns only metadata the SPA needs to render buttons:

```json
{
  "methods": [
    { "type": "password" },
    {
      "type": "oidc",
      "label": "Acme Corp SSO",
      "loginPath": "/api/v1/auth/oidc/login/<env-uuid>"
    }
  ]
}
```

`config` is never returned over the wire — only `type` and `label`.

### State cookie

Generated by `pkg/auth/state.go`:

```go
func IssueStateCookie(w http.ResponseWriter, jwtSecret []byte, state State, ttl time.Duration) error
func ParseStateCookie(r *http.Request, jwtSecret []byte) (State, error)
```

State is a JWT signed with HS256 using the existing api JWT secret. Claims:
```json
{
  "env": "<env-uuid>",
  "nonce": "<random>",
  "verifier": "<pkce verifier or empty>",
  "exp": "<now + 10 min>",
  "iss": "osctrl-api",
  "aud": "osctrl-auth-state"
}
```

`aud=osctrl-auth-state` so a state JWT can't be used as a user-auth JWT and
vice versa (they share a signing key; the audience claim segregates them).

Cookie: `osctrl_auth_state`, HttpOnly, Secure, SameSite=Lax, Path=`/api/v1/auth/`,
10-minute Max-Age.

### Callback → JWT issuance

Generic flow in `cmd/api/handlers/auth_callback.go`:

1. Parse `{type}` from path, validate it's a known provider type
2. Parse state cookie → `State{EnvUUID, Nonce, Verifier}`
3. Load env from `EnvUUID`; permission-check the env is enabled
4. Load provider from DB by `(env.ID, type)`; decrypt secrets
5. Construct provider via `pkg/auth/oidc.NewOIDCProvider(decryptedCfg)`
6. Call `provider.HandleCallback(ctx, r)` → `ResolvedIdentity`
7. Resolve user via `ResolveOrJIT(identity, env)`:
   - existing username matches → use that AdminUser
   - else if `jit_provision: true` → create AdminUser (no env permissions)
   - else → reject
8. Mint JWT via `users.UserManager.CreateToken(...)` (the WithJWT path from #813)
9. Set `osctrl_token` HttpOnly Secure cookie
10. Audit-log via `auditLog.NewLogin(user.Username, utils.GetIP(r))`
11. `http.Redirect(w, r, "/", http.StatusFound)`

The handler is **protocol-agnostic** — step 5 picks the right concrete
provider, step 6 returns a protocol-neutral identity, the rest is shared.

### Login UX in SPA

`frontend/src/routes/login.tsx`:

```tsx
// 1. User picks env
// 2. SPA calls GET /api/v1/login/{env}/methods → renders options:
//    - If methods includes 'password': show username/password form
//    - For each non-password method: render a button
//      <Button onClick={() => window.location.href = m.loginPath}>
//        Sign in with {m.label}
//      </Button>
// 3. User clicks SSO button → full-page nav to /api/v1/auth/oidc/login/{env}
// 4. Provider handles the dance
// 5. Browser lands back on / with osctrl_token cookie set
// 6. TanStack Query refetches; user is in
```

No special SPA logic per provider type — the methods endpoint is the
single source of truth for what's available.

## Testing strategy

### Unit
- `pkg/auth`: state JWT encode/decode (happy path + tampered + expired + wrong audience).
- `pkg/auth/oidc`: claim resolution (`pickUsername`, `hasRequiredGroup`).
- `pkg/environments`: AuthProviders CRUD + secret encryption round-trip.

### Integration (against real Keycloak on Kali)
- Methods endpoint returns the expected shape for an env with no provider, password-only, OIDC-only.
- Login redirect produces a valid Keycloak authorize URL with correct state.
- Callback exchanges code for tokens, verifies id_token, resolves alice, issues osctrl JWT cookie.
- JIT provisioning creates the AdminUser exactly once across multiple logins.
- Required-group gate rejects users not in the configured group.
- State-cookie tampering rejected (alter the JWT, expect 403).
- State-cookie cross-env replay rejected (state issued for env A used in callback for env B — env claim mismatch).

### Cross-IdP (Keycloak + second IdP, e.g., Auth0 free tenant)
- Same flow against Auth0 to confirm we don't accidentally rely on Keycloak-specific claim shapes.
- Compare emitted claims, document any differences in the README.

## Security: threat model + pentest plan

Auth surfaces are the highest-stakes part of the system. A gap here means an
attacker becomes super-admin of every customer's fleet — the kind of impact
that can't be patched after the fact. This section is mandatory; the feature
is not "done" until the pentest passes.

### Threat model — attackers we defend against

1. **Unauthenticated internet attacker** hitting `/api/v1/auth/oidc/login/*`
   and `/api/v1/auth/oidc/callback` directly with crafted inputs.
2. **Logged-in low-privilege user** trying to escalate to admin or access
   other envs' data via auth flows.
3. **Compromised IdP** issuing assertions/tokens for users that shouldn't
   exist (or for legitimate users with wrong claims). osctrl-api must not
   blindly trust everything the IdP says.
4. **Compromised browser session** (XSS, malicious extension) trying to
   exfiltrate the osctrl_token cookie or the state cookie.
5. **Network attacker** on the path between osctrl-api ↔ IdP (rare but real
   for on-prem deploys without TLS to internal IdPs).
6. **Insider with DB access** (read-only) trying to extract OIDC client
   secrets or forge auth state.
7. **Tenant-to-tenant escalation** in the SaaS model: tenant A's operator
   tries to authenticate against tenant B's env.

### Threats catalogued (numbered for traceability)

| # | Class | Specific threat | Mitigation |
|---|---|---|---|
| T1 | Auth bypass | id_token signature not verified → attacker signs their own token | `gooidc.Verifier.Verify` is the only way to consume id_token; never `Claims()` raw |
| T2 | Auth bypass | iss claim not verified → token from a different IdP accepted | `gooidc.NewProvider` pins issuer; `Verify` checks it |
| T3 | Auth bypass | aud claim not verified → token issued to a different client accepted | `gooidc.Config{ClientID}` pins audience |
| T4 | Auth bypass | exp not verified → expired token replayed | `Verify` checks `exp` (clock skew tolerance 0 or small) |
| T5 | Auth bypass | nbf not verified → pre-valid token accepted | `Verify` checks `nbf` if present |
| T6 | CSRF | callback called without prior login → attacker tricks victim's browser into authenticating as attacker | `state` cookie required and matched against URL state param; HttpOnly so JS can't forge |
| T7 | CSRF | state value guessable | 256-bit cryptorandom from `crypto/rand` |
| T8 | Replay | code reused across multiple callbacks | Single-use guaranteed by IdP; verify exchange returns OK once per code |
| T9 | Replay | state reused across multiple callbacks | State cookie cleared after use; subsequent callback gets "missing state" |
| T10 | PKCE bypass | verifier omitted but expected | If `usePKCE: true` and verifier missing in state, reject |
| T11 | Token disclosure | id_token logged or returned in URL | Never log raw tokens; never put in URL; only set via Set-Cookie |
| T12 | Token disclosure | id_token in browser history via fragment | Use HttpOnly cookie, not URL fragment or query |
| T13 | Open redirect | post-login redirect to attacker-controlled URL via `?next=` | Allowlist redirect targets to same-origin paths only; no off-host redirects |
| T14 | Open redirect | IdP's `redirect_uri` not validated against allowlist on osctrl side | Provider config locks `redirect_url` to one value; never read from request |
| T15 | Username spoofing | preferred_username collides with existing local user, OIDC user takes over their account | On JIT, if AdminUser exists with same username but no link to OIDC identity, **reject** rather than overwrite |
| T16 | Privilege escalation via JIT | Attacker creates IdP user, gets JIT'd into osctrl with broad perms | v1: JIT creates user with zero permissions; operator grants manually. Document this clearly. |
| T17 | Group claim spoofing | Attacker IdP returns `groups: ['admin']` | RequiredGroups gate validates against env config; group claim doesn't grant osctrl permissions directly in v1 |
| T18 | Cross-env auth | State for env A used to log into env B → user gets perms they shouldn't | State JWT contains `env` claim; callback verifies it matches the env in the callback URL |
| T19 | Token confusion | osctrl JWT and auth state JWT share signing key | Different `aud` claim (`osctrl-api` vs `osctrl-auth-state`); verifier enforces audience |
| T20 | Secret disclosure | OIDC client_secret in plaintext in DB or logs | AES-GCM at rest; never logged; never returned in API responses |
| T21 | Secret disclosure | client_secret leaked via error messages | Generic error responses; raw provider errors logged server-side only |
| T22 | Discovery poisoning | Attacker MITMs the OIDC discovery endpoint | osctrl-api connects to IdP over HTTPS only; verify cert; document HTTP-IdP only allowed in dev mode |
| T23 | Code injection in claims | preferred_username contains shell metacharacters / SQL / XSS payload | Username goes through `users.EnvNameFilter`-equivalent regex before DB write |
| T24 | Email/username substitution | IdP returns mutable email; user changes email to victim's, takes over their account | Username is the stable identifier (`sub` preferred). If `email` is the username claim, document the trust requirement clearly |
| T25 | Mass-assignment via JIT | JIT-created AdminUser ends up with `Admin: true` somehow | JIT path explicitly constructs AdminUser with `Admin: false, Service: false`; never deserialize directly from claims into the struct |
| T26 | Audit-log poisoning | Identity fields (username, email) contain newlines/control chars that confuse log parsers | Sanitize before audit-log write (strip non-printables) |
| T27 | Rate-limit bypass | Attacker brute-forces state tokens via /callback | State JWT is HMAC-signed with the api's JWT secret; forgery requires the secret |
| T28 | DoS via state-cookie flood | Attacker spams /auth/oidc/login/{env} generating throwaway state cookies | Rate-limit /api/v1/auth/* per IP (reuse pkg/ratelimit) |
| T29 | Method enumeration | /login/{env}/methods reveals which envs have OIDC configured | Acceptable — same info as the SPA's UI; not a secret |
| T30 | Tenant enumeration via /methods | Attacker enumerates valid env UUIDs by 404 vs 200 on /methods | Return 200 for any well-formed UUID with empty methods array if env doesn't exist or no providers; envs-list endpoint already provides this info to authenticated users |
| T31 | Timing oracle on state verification | Attacker uses response timing to find valid state JWTs | constant-time HMAC verify (Go's stdlib does this) |
| T32 | Token theft via XSS in SPA | XSS in any SPA route can read osctrl_token | osctrl_token is HttpOnly (already so per #813); attacker can use the cookie but can't exfiltrate. CSRF mitigation via SameSite=Lax + double-submit token. |

### Hardening rules — non-negotiable for v1

These get encoded into the implementation and the test suite verifies each:

- **No raw token logging**. `log.Err(err).Msg(...)` where `err` may contain
  token bytes is forbidden. Use structured logging with explicit fields and
  redact token values.
- **No raw claims logging at INFO level**. Email and PII go to DEBUG only.
- **No off-host redirects on login completion**. `next` param (if added)
  validated against an allowlist of relative paths; reject any absolute URL.
- **No HTTP IdPs in production**. Dev mode (`-e OSCTRL_DEV_ALLOW_HTTP_IDP=1`)
  required to use a `http://` issuer URL. Default rejects it.
- **No client-secret retrieval via API**. The /environments endpoint never
  returns `client_secret`, even to super-admin. Operators write it via PATCH
  but can't read it back. Reset via "rotate secret" admin action.
- **No JIT auto-promotion**. Even if all the stars align, JIT cannot create
  an AdminUser with `Admin: true` or `Service: true` or any permissions
  beyond what's explicitly granted. Compile-time enforced via a
  `JITNewUser(username, email, name)` constructor that doesn't take those
  fields.
- **Single-use state**. State cookie's underlying nonce is recorded
  ephemerally (in-memory or Redis with TTL = state TTL) and rejected on
  reuse. Belt-and-braces beyond the JWT exp.

### Pentest plan (mandatory)

The pentest runs **before any push to origin/main**, and again **before any
upstream PR**. Two passes:

#### Pass 1: focused vector tests (mandatory; gates the merge)

A scripted test suite under `pkg/auth/security_test.go` and
`cmd/api/handlers/auth_security_test.go` that exercises each numbered threat
T1..T32. Each test produces a clear pass/fail. The suite must be 100% green.

#### Pass 2: scenario-based pentest against a live Kali dev stack

Run a 16-scenario adversarial playbook. Each scenario is a sequence of
HTTP requests / browser actions / DB inspections designed to verify that
the system fails closed under realistic attack patterns:

1. **Anonymous direct hit on /callback** with no state cookie → expect 403.
2. **State cookie alteration** (flip one base64 char in the JWT) → expect 403.
3. **State cookie from yesterday** (resign expired-exp claim) → expect 403.
4. **State cookie with wrong audience** (`aud: osctrl-api` instead of `osctrl-auth-state`) → expect 403.
5. **State cookie for env A used in callback for env B** → expect 403 (env claim mismatch).
6. **Code parameter omitted** → expect 403, no DB writes.
7. **Code parameter with random junk** → expect 403, no DB writes, no provider call leaked sensitive error.
8. **Replay the same callback URL twice** → second call expect 403 (state cleared).
9. **Stale id_token** (capture one, wait past exp, replay) → expect 403.
10. **Forged id_token** (sign with attacker's key, same iss claim) → expect 403 (signature fail).
11. **id_token for a different audience** → expect 403.
12. **id_token with `preferred_username` containing newline and SQL injection** → expect rejection at user-resolve, no DB write.
13. **Existing osctrl password user "alice" + JIT'd "alice" from OIDC** → second login expects rejection (T15: account-takeover prevention).
14. **JIT-created user calls /api/v1/environments** → expect empty list (zero perms granted).
15. **DB inspection: client_secret column never readable** as plaintext.
16. **Browser inspection: osctrl_token cookie is HttpOnly+Secure+SameSite=Lax**, state cookie similar with path scoping.

The 16 scenarios are documented in `docs/security/auth-pentest.md` (added in
this PR) with the curl/playwright invocations for each. Result of every
scenario captured in the PR description as evidence.

#### Pass 3: external review (recommended, not blocking)

After Pass 1+2 are green, share the spec + diff with at least one external
security-minded reviewer (or the security-auditor subagent). Pass-3 is for
defense-in-depth, not a gate.

- **Manual curl scripts** (most pentest scenarios are HTTP-level)
- **`go vet` and `staticcheck`** for static issues
- **`gosec`** for security-focused static analysis (we've used it in prior rounds)

### What "flawless" means

The PR is not merge-ready until **every one of the following is true**:

- [ ] All 32 numbered threats have an explicit test or design rationale
- [ ] All 16 pentest scenarios pass on the Kali dev stack
- [ ] `gosec ./pkg/auth/... ./cmd/api/handlers/auth_*` → zero HIGH or MEDIUM findings
- [ ] `govulncheck ./...` → zero CVEs in the auth code path
- [ ] No raw secrets, no raw tokens, no PII at INFO level in any log line
- [ ] Cross-IdP test (Keycloak + Auth0) demonstrates the same behavior
- [ ] One external review pass on the diff

This is what "we can't leave security gaps here" looks like operationally.
The cost is real (probably +2 days for the full pentest suite + +1 day for
external review feedback) but it's bought once and amortized over every
customer who runs osctrl federated against their IdP.

## What this design does NOT do (and why)

- **Per-tenant config** (in the multi-tenant SaaS sense): per-env is enough today. Multi-tenant SaaS uses one osctrl deployment per tenant in the planned silo architecture, so per-env *is* per-tenant. If we ever move to a shared multi-tenant osctrl, the `env_auth_providers` table simply gains a `tenant_id` column.
- **Login policy DSL**: no "if user has group X give them permissions Y" rules in v1. The legacy-admin-equivalent (zero permissions on JIT) is honest about what we know. Extension point is `ResolveOrJIT(identity, env) (AdminUser, error)` — when we add policy, it goes there.
- **IdP-initiated login**: not in v1 for either OIDC or SAML. Adds complexity (URL parameter validation, env disambiguation) for marginal user benefit.
- **Account linking**: if a user logs in via OIDC with username "alice" but also has an osctrl password-auth user "alice", v1 just uses the matching row. No separate "OIDC identities" table. If conflicts emerge in practice we add linking.

## Backwards compatibility for cmd/admin

The `auth/01-oidc-provider` branch refactors legacy admin to consume
`pkg/auth/oidc`. Compatibility audit:

### Preserved (zero behavior change)

- YAML config schema (`YAMLConfigurationOIDC`): unchanged.
- CLI flags: unchanged.
- Routes (`/login`, `/oidc/callback`): unchanged. Path constants
  (`oidcCallbackPath = "/oidc/callback"`) preserved verbatim so
  existing IdP-side redirect-URI registrations don't break.
- Session cookie (post-login): unchanged.
- `pkg/users` and session-manager API: unchanged.
- DB schema: unchanged (this branch).
- JIT-provisioning policy (zero env permissions on first login).
- Required-groups gate semantics.

### Breaks made backwards-compatible via shim

- **Username regex**: pre-refactor cmd/admin didn't validate
  `preferred_username` against any character class. The new
  `pkg/auth/oidc.sanitizeUsername` enforces `[a-zA-Z0-9_-]{1,64}`.
  Deployments where the IdP emits email-format usernames (e.g.,
  `alice@example.com`) would break under strict validation.
  Mitigated by `Config.LegacyPermissiveUsername`: when true, the
  regex is bypassed but TrimSpace + empty-check stay (audit-log
  poisoning T26 partially defended). cmd/admin sets it true;
  cmd/api leaves it false (strict default).

### Accepted minor break

- **State cookie name** changes from `osctrl-admin-oidc-state` (encoded
  via gorilla/securecookie) to `osctrl_auth_state` (HMAC JWT).
  Operators mid-OIDC-login at the exact moment of a deploy will
  see one "forbidden" redirect and need to retry once. Self-healing
  on retry. No data corruption, no security implication, no
  long-lived effect. Documented here so it's not a surprise.

## Risks and open questions

1. **Encryption key management**: deriving the secret-encryption key from the JWT secret is convenient but means rotating the JWT secret invalidates all stored OIDC client secrets. Real fix is a separate `OSCTRL_SECRET_KEY` env var. v1 uses JWT-derived for simplicity; document the limitation.
2. **Upstream acceptance**: javuto might prefer global config over per-env. Design supports both — per-env is a strict superset. If he pushes back, we can offer to collapse to global before merge.
3. **Browser cookie behavior**: the state cookie path is `/api/v1/auth/` so it doesn't leak to other endpoints. SPA cookies (`osctrl_token`) live at `/`. Different paths means the browser sends each only where appropriate.
4. **Test IdP cost**: Keycloak is free. Auth0 free tier is sufficient. Beyond that, testing against Entra/Google/Okta needs accounts. Defer until a customer-driven need exists.

### Manual smoke checklist (happy path)

The credential-driven happy path requires driving Auth0's New
Universal Login, which is client-side React not amenable to scripting
over HTTP. Validate manually via a real browser:

1. Browse to `http://192.168.31.239:9100/api/v1/auth/oidc/login`.
   Browser redirects to Auth0 login page.
2. Sign in as `alice@osctrl.local` / `AliceTestPw-2026!`.
3. Auth0 redirects back to `/api/v1/auth/oidc/callback?code=…&state=…`.
4. osctrl-api callback sets `osctrl_token`, `osctrl_csrf`, AND
   `osctrl_id_token` (the new id_token_hint cookie).
5. Browser navigates to `/` and the SPA shows alice's username.
6. `alice` row exists in `admin_users` with `auth_source='oidc'`,
   `admin=false`, `service=false`.
7. Sign out → SPA navigates to
   `https://dev-…us.auth0.com/oidc/logout?post_logout_redirect_uri=…
    &id_token_hint=eyJ…&client_id=A4unJh…`. Auth0 terminates the
   session and redirects back to `/login` without an error.
8. Click "Continue with SSO" again → Auth0 prompts for credentials
   (no silent re-auth).

### Auth0-specific notes

- **Strict username regex blocks Auth0's default `sub`.** Auth0 issues
  subjects like `auth0|6a0a4280eb13abdd6cfa2fc1` containing the `|`
  character, which our regex rejects. Operators using Auth0 with
  `cmd/api` must set `--oidc-username-claim nickname` (or
  `preferred_username` once they've configured their tenant to emit
  that claim). Tested with `nickname` → maps to `alice` cleanly.

- **Groups claim requires an Auth0 Action.** Auth0 does NOT emit a
  `groups` claim by default. To use `--oidc-required-groups`,
  operators add a post-login Action (we deployed
  `add-osctrl-groups-claim` for the pentest pass). The Action reads
  `event.user.app_metadata.groups` and writes the claim. Operators
  using SAML→Auth0 federation get groups for free from the upstream
  IdP.

- **id_token_hint required for logout.** Auth0's `/oidc/logout`
  endpoint REJECTS `?post_logout_redirect_uri=...` without
  `id_token_hint` ("Missing parameter: id_token_hint"). This drove
  the auth/04 commit that persists the raw id_token in an HttpOnly
  `osctrl_id_token` cookie and returns it via `/logout`. Keycloak
  accepts `client_id` as an alternative; we send both.

- **Cross-origin auth requires HTTPS.** Auth0's `/co/authenticate`
  password endpoint (used by single-page apps) only works over HTTPS
  origins. Our dev test environment is plain HTTP, which is why the
  happy path is manual rather than scripted. Production deployments
  must front osctrl-api with TLS regardless (the `Secure` flag on
  every cookie already mandates this).

- **Default signing alg is HS256 — must be switched to RS256.**
  Contrary to first impression, Auth0's "Regular Web Application"
  template DEFAULTS to HS256 (symmetric, using client_secret as the
  HMAC key). `go-oidc` rejects HS256 because the verifier checks
  signatures via JWKS, not a shared secret — and that rejection is
  correct (HS256 means client compromise == arbitrary token mint).
  Operators MUST switch the app's "JsonWebToken Signature Algorithm"
  to RS256: Applications → osctrl-api → Advanced Settings → OAuth
  → Algorithm = RS256. Via API: `PATCH /api/v2/clients/{id}` with
  body `{"jwt_configuration":{"alg":"RS256"}}`. Failure mode without
  this: `oidc: id_token verification failed: oidc: malformed jwt:
  unexpected signature algorithm "HS256"`.

## SAML 2.0 provider design (`auth/05-saml-provider`)

OIDC ships in `auth/00`–`auth/04`. SAML 2.0 Web Browser SSO Profile
is the second federated protocol we support. This section captures
the design decisions, the threat catalogue, and the deferred-to-v2
list before implementation begins.

### Library choice

`github.com/crewjam/saml v0.5.1` — already pinned in `go.mod` and
used by `cmd/admin` for legacy SAML support. Battle-tested in many
Go services, MIT-licensed, handles XMLDSig correctly. We use the
lower-level `samlsp.ServiceProvider` directly (NOT the high-level
`samlsp.New()` middleware) because cmd/api issues its own JWT
cookies after the SAML round-trip — parallel to how the OIDC
provider does it, and parallel to the existing protocol-neutral
`auth_resolve.go` + `auth_jwt.go` helpers.

### Design decisions

**D1. Cookie strategy.** Reuse the existing `pkg/auth.State` cookie
(HMAC JWT, audience `osctrl-auth-state`) for CSRF defense. The
nonce field is round-tripped through SAML's RelayState, verified
on the ACS callback. Mints the same `osctrl_token` + `osctrl_csrf`
cookies after success — the SPA's auth-check logic doesn't care
which protocol generated the session. `osctrl_id_token` (used for
OIDC RP-initiated logout) is not applicable to SAML; the cookie is
skipped for SAML logins and the `LogoutResponse.IdPIDTokenHint`
field stays empty.

**D2. SP-initiated only in v1.** IdP-initiated flows (where the
user starts at a corporate portal and is POSTed directly to our
ACS) are convenient for some operators but break the InResponseTo
correlation we rely on for replay defense. Documented limitation;
operators with corporate portals fall back to bookmarking the SP-
initiated login URL.

**D3. No Single Logout (SLO) in v1.** SAML's SLO is famously broken
across IdPs — half the implementations don't support it correctly
and the ones that do disagree on the bindings. Logout on the SP
clears osctrl cookies + APIToken; the IdP session keeps running
until its own timeout. Documented limitation; operators concerned
about IdP session lifetime tune the IdP-side timeout.

**D4. SP metadata served at `/api/v1/auth/saml/metadata`.** GET
returns the SP-side XML metadata (entity ID, ACS URL, certificate
descriptors). IdP admins paste this URL into their IdP-side SP
config; the IdP auto-discovers our endpoints. Without this, every
IdP admin would hand-craft the SP config.

**D5. No AuthnRequest signing in v1.** Signing requires us to ship
an SP signing cert+key pair, which is operationally annoying for
deployments. The library default (unsigned AuthnRequests, signed
responses required) is sufficient — an attacker who forges an
AuthnRequest still needs the IdP's private key to forge the
signed response.

**D6. Global, single IdP per deployment.** Same shape as OIDC:
one SAML IdP configured via flags/YAML, applies deployment-wide.
Per-env IdP routing is deferred to v2 (same rationale as the OIDC
pivot).

**D7. AuthMethods response.** `/api/v1/auth/methods` returns a list
of objects each with `type` ("password" | "oidc" | "saml") and
`loginUrl`. SPA renders one button per advertised type. For v1 the
labels are hardcoded ("Sign in", "Continue with SSO (OIDC)",
"Continue with SSO (SAML)"); a per-provider `displayName` field
can be added later if operators ask for it.

### Configuration surface

`pkg/config.YAMLConfigurationSAML` (already exists for cmd/admin;
extended here):

| Field | Purpose |
|---|---|
| `Enabled` | Gate that turns on the routes + Init at startup |
| `IDPMetadataURL` | URL to IdP-side XML metadata document |

### Threat catalogue (SAML-specific, layered onto OIDC's T1–T32)

| ID | Threat | Defense |
|----|--------|---------|
| **S1** | **XML Signature Wrapping (XSW)** — assertion forgery where the legitimate signature is preserved but the verifier reads claims from a different node than the one the signature covers. The category-killer of SAML implementations. | `crewjam/saml`'s reference resolution verifies signature against the SAME node that claims are extracted from. Regression test constructs XSW payloads (8 known variants) and asserts each is rejected. |
| **S2** | **Unsigned assertion accepted.** | `RequireAssertionSigned=true` (always on). Probe: strip `<ds:Signature>` from a valid response → reject. |
| **S3** | **Assertion replay** within validity window. | crewjam ships a `ReplayDetector` (LRU on assertion ID). We enable it with a 24h window. Probe: replay the same SAMLResponse → second time rejected. |
| **S4** | **Audience confusion** — assertion meant for a different SP. | `AudienceRestriction` checked against our EntityID. Probe: forge `<saml:Audience>` to a different SP → reject. |
| **S5** | **Clock-skew abuse** — assertion outside NotBefore/NotOnOrAfter window. | Library check with ±60s default; we set `ReplayWindowMinutes=5`. Probe: replay an assertion 10 minutes old → reject. |
| **S6** | **SubjectConfirmationData.Recipient mismatch** — relay attack where assertion meant for SP A gets POSTed to SP B. | Library verifies Recipient against ACS URL. Probe: POST SAMLResponse with `Recipient=http://attacker.example/` → reject. |
| **S7** | **InResponseTo replay/forgery** — SP-initiated flow's response correlation. | RelayState cookie carries the AuthnRequest ID; ACS verifies the assertion's `InResponseTo` against the cookie. Probe: POST ACS with no prior cookie → reject. |
| **S8** | **Algorithm downgrade** — SignatureMethod set to a weak alg (MD5, SHA1). | Library rejects weak algs by default. We pin RSA-SHA256 minimum via `SignatureMethod` config. Probe: forge SignatureMethod="...rsa-sha1" → reject. |
| **S9** | **XXE in XML parser.** | Go's `encoding/xml` is XXE-immune by default. crewjam uses standard parsers. Probe: SAMLResponse with `<!DOCTYPE ...>` entity → reject (parser ignores entities). |
| **S10** | **RelayState injection** — open-redirect or audit-poison via the server-opaque RelayState parameter. | RelayState carries our HMAC `auth.State` JWT (audience `osctrl-auth-state`). Garbage RelayState fails JWT parse; tampered RelayState fails HMAC; legit RelayState that doesn't match the cookie's nonce is rejected like OIDC's T6 CSRF. Probe: forge RelayState with attacker-chosen nonce → reject. |
| **S11** | **Encrypted assertions** (deferred). | Not supported in v1. Operators requiring `<saml:EncryptedAssertion>` (some private-cloud IdPs) must keep using legacy admin until v2 ships encryption support. Documented limitation. |

Plus the OIDC-shared threats that map cleanly:
- T6 CSRF → RelayState round-trip (replaces OAuth2 state)
- T9 single-use cookie → ClearStateCookie on ACS
- T17 group gate → reads `GroupsAttribute` from AttributeStatement
- T18 cross-env replay → RelayState carries env UUID, ACS verifies
- T23 username injection → sanitizeUsername reused
- T26 audit-log poisoning → IdP `<samlp:StatusMessage>` never echoed
- T31 timing-oracle → every rejection redirects to `/` with no detail

### Implementation plan

**Step 1 — `pkg/auth/saml/` package** (~400 LOC + tests):
- `config.go` — Config struct, `Validate()`.
- `provider.go` — Provider implementing `auth.Provider`. Internally
  builds a `samlsp.ServiceProvider`. `LoginURL` returns IdP SSO URL
  with `RelayState=<state cookie value>`. `HandleCallback` parses
  the POST body's `SAMLResponse`, calls `sp.ParseResponse`, extracts
  username + groups, returns `auth.ResolvedIdentity`.
- `claims.go` — `pickSAMLUsername(assertion, attributeName)`,
  `hasSAMLRequiredGroup(assertion, attribute, required)`.
- Tests: XSW regression suite, replay test, audience test, recipient
  test, alg-downgrade test, unsigned-assertion test. crewjam's
  internal test helpers provide a working signer.

**Step 2 — `cmd/api/handlers/auth_saml.go`** (~180 LOC):
- `InitSAML(ctx, cfg)` — builds provider, stashes in package global.
- `SAMLLoginHandler` (GET) — mints state cookie, redirects.
- `SAMLACSHandler` (POST) — parses + verifies, resolves user via
  `resolveFederatedUser`, issues session JWTs via
  `userJWTSessionTokens`. Mirror of `OIDCCallbackHandler` but for
  the POST-body wire shape.
- `SAMLMetadataHandler` (GET) — returns SP metadata XML.

**Step 3 — `cmd/api/main.go` routes**:
- `GET  /api/v1/auth/saml/login`     (loginRateLimit)
- `POST /api/v1/auth/saml/acs`       (preAuthRateLimit, NO CSRF — IdP doesn't have our token)
- `GET  /api/v1/auth/saml/metadata`  (preAuthRateLimit)

**Step 4 — `cmd/api/handlers/auth_methods.go`** — append SAML
method when configured.

**Step 5 — `pkg/config/types.go` + `flags.go`** — extend
YAMLConfigurationSAML, add `initSAMLFlags(params)` for cmd/api.

**Step 6 — SPA `LoginPage.tsx`** — render SAML button when present
in `/auth/methods` response.

**Step 7 — Live pentest**:
- Add SAML client to Keycloak's `osctrl` realm via admin API.
- Drive the negative-path probes through curl/python (same shape
  as OIDC pentest pass).
- Document results in this spec as "Pentest pass 5: SAML
  (Keycloak)" and "Pentest pass 6: SAML (Auth0)".

### What's deferred to v2

- Encrypted assertions (`<saml:EncryptedAssertion>`).
- IdP-initiated flows (no AuthnRequest from us).
- Single Logout (SLO).
- AuthnRequest signing (SP private key + cert).
- Multiple SAML providers per deployment.
- Per-env IdP routing (matches the OIDC v1 limitation).
